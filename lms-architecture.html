System Overview & Design Principles

🧠
Agent-First Architecture
Every workflow is orchestrated by an AI agent layer that understands intent, breaks tasks into tool calls, and manages multi-step execution with memory continuity.
⚡
Event-Driven Everything
Services communicate via Kafka event streams — no tight coupling. Any service can react to any event. New integrations wire in without touching existing code.
🗣️
Conversational Admin Studio
Admins chat with the LMS. Upload a PDF syllabus → AI creates full course, quiz bank, certificates, and badge criteria in minutes. No UI navigation required.
🔌
Hot-Pluggable Integrations
External tools (Zoom, Stripe, Slack, SCORM) register in the Dynamic Tool Registry at runtime. No redeployment. Admin toggles tools on/off from dashboard.
// CORE DESIGN AXIOMS 1. Every component is a microservice with a single responsibility 2. All inter-service communication is async via Kafka OR sync via gRPC (never REST internally) 3. Agent Orchestrator is the ONLY stateful coordinator — all services are stateless 4. Memory is tiered: Hot (Redis) → Warm (PostgreSQL) → Cold (S3/MinIO) → Semantic (pgvector/Weaviate) 5. Every event is audited, signed, and stored immutably in ClickHouse 6. Security is enforced at API Gateway + service mesh (mTLS) — not inside services 7. Tool Registry is the single source of truth for all AI tool capabilities 8. Admin chat is a first-class citizen — not a bolt-on feature

Admin Chat → Action Pipeline

01
Input Reception & File Processing
Admin message + optional file uploads received via WebSocket. Files stored in MinIO, text extracted via Tika/PyMuPDF, chunked and embedded via sentence-transformers. File metadata stored in PostgreSQL.
WebSocket Apache Tika PyMuPDF MinIO
02
Intent Classification & Planning
LLM (GPT-4o / Claude 3.5) classifies intent, extracts entities, generates execution plan as a DAG of tool calls. Plan stored in Redis for resumability. Context pulled from all memory layers.
LangGraph GPT-4o Redis DAG MCP
03
Tool Dispatch & Parallel Execution
Orchestrator dispatches parallel tool calls to Course Service (create modules), Quiz Engine (generate questions from LO embeddings), Badge Engine, Certificate Engine. Each tool call is tracked with correlation IDs.
Tool Registry gRPC parallel Correlation IDs Temporal.io
04
Content Generation & AI Enrichment
Quiz questions generated from extracted learning objectives + knowledge base. Badge criteria synthesized from completion requirements. Certificate template personalized. All content stored with version history.
GPT-4o pgvector RAG Weaviate PDF Gen
05
Event Publication & Integration Triggers
All created artifacts emit Kafka events (course.created, quiz.created, badge.configured, certificate.template.ready). Integration workflows (Slack, email) are triggered asynchronously without blocking the response.
Apache Kafka Avro Schema Temporal Webhooks
06
Streamed Response & Audit
Results streamed back to admin via SSE/WebSocket with live progress. Full audit log written to ClickHouse: who requested what, which tools ran, what was created, timestamps. Admin can preview, edit, or rollback.
SSE Streaming ClickHouse Audit Version Control Rollback

Central brain. Receives all admin/user intent, maintains conversation state, executes multi-step plans via tool calls, reads/writes all memory layers.

  • Context window management (MCP protocol)

  • Conversation memory persistence

Course Service
Manages the full course lifecycle — creation, versioning, publishing, enrollment, progress tracking, SCORM compatibility.

AI-powered quiz generation, delivery, and analysis. Adapts difficulty in real-time based on learner performance using IRT (Item Response Theory).

Multi-Layer Memory Architecture

⚡
Working Memory
Redis Cluster
Active conversation state, current tool execution plan, session data, rate limit counters, pub/sub channels. TTL: 24 hours. Sub-millisecond access.
🐘
Episodic Memory
PostgreSQL + pgvector
Full conversation history, user learning history, course content, quiz results, certificates. Embeddings stored via pgvector for semantic retrieval. The source of truth.
🔷
Semantic Memory
Weaviate
Course content embeddings, skill concept vectors, learning objective embeddings. Powers semantic search, personalization, and RAG-based quiz generation.
🌐
Context Protocol
MCP (Model Context Protocol)
Structured context exchange between AI models and tool registry. Maintains tool schemas, current permissions, and domain knowledge in a standardized format readable by any LLM.

Enterprise Observability Stack

The Three Pillars + AI

📊 METRICS
Prometheus scrapes all services every 15s. Custom metrics: agent_tool_calls_total, course_creation_duration_ms, quiz_generation_latency_p99, certificate_issuance_errors_total. Grafana dashboards per service + exec summary.
🔍 DISTRIBUTED TRACING
OpenTelemetry SDK in every service. Jaeger/Tempo collects spans. Every admin chat message gets a trace_id that follows all tool calls across services. P99 latency SLAs enforced via alerts.
📝 LOGS
Structured JSON logs via Fluentd → Loki. PII automatically redacted (email, names). Log retention: 90 days hot, 2 years cold (S3). Alerting on error rate spikes via Grafana OnCall.
🤖 AI OBSERVABILITY
LLM call tracking: model, tokens in/out, latency, cost per call. Tool call success/failure rates. Agent plan quality scoring. Anomaly detection on AI cost spikes. Langfuse integration for LLM ops.

Architecture Summary

This blueprint represents a production-grade, cost-optimized, AI-native LMS designed to scale from 100 to 1M+ learners without architectural changes. Every component is designed for independent deployment, scaling, and replacement.

  • Gateway: Kong / Traefik
  • Message Bus: Redpanda (Kafka-compat)
  • Primary DB: PostgreSQL + Patroni
  • Vector DB: Weaviate + pgvector
  • Cache: Redis / Dragonfly
  • Analytics: ClickHouse
  • Auth: Keycloak + OPA
  • Service Mesh: Istio + Envoy
  • GitOps: ArgoCD + Helm
